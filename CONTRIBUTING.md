# Contributing to DesignBench

## Workflow

1. Start with or link to an issue.
2. Clarify scope and acceptance criteria before implementation.
3. Keep pull requests focused and reviewable.
4. Update documentation when behavior, structure, or decisions change.
5. Record major technical decisions in `docs/adr/`.

## Pull requests

Each pull request should:

- reference the related issue
- describe what changed
- avoid unrelated changes
- update docs where appropriate

## Documentation

Documentation in this repository is part of the source of truth for the product.

Prefer:
- Markdown in version control
- explicit architecture notes
- ADRs for major decisions
- clear domain terminology
